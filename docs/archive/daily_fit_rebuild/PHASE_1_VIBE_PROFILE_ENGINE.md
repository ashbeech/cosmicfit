# Phase 1: DailyEnergyEngine — Vibe Profile Generation

**Dependency:** Phase 0 (all types from `DailyFitTypes.swift` must exist and compile).
**Produces:** The vibe profile generation core of `DailyEnergyEngine` — takes raw astrological inputs and outputs a `VibeBreakdown` (6 energies, 21 points).
**Estimated scope:** ~200–280 lines of engine code. One new file.

---

## 1. Context

In the legacy system, vibe profiles are generated by a chain of:

1. `SemanticTokenGenerator.generateDailyFitTokens()` (4,123 lines) — produces 80–150 `StyleToken` objects from natal chart, transits, moon phase, progressed chart, and weather.
2. `VibeBreakdownGenerator.generateVibeBreakdown()` (887 lines) — maps those tokens to 6 energy buckets via hardcoded string sets, then normalises to 21 points.

This 5,000-line pipeline is being replaced by a single focused engine. Your job is to build the **vibe profile generation** portion of `DailyEnergyEngine`. This phase does NOT handle axes, transit summaries, or lunar context — those come in Phase 2.

### Why the legacy approach fails

1. **String-matching fragility.** Token names are freeform strings. The energy mapping checks `token.name.lowercased()` against hardcoded `Set<String>` (e.g. `classicTokens`, `dramaTokens`). If the generator produces "Structured" instead of "structured", the mapping silently fails. Overlapping sets mean tokens like "bold" map to both Drama and Playful for Leo.
2. **Inconsistent weight scales.** `WeightingModel` has `natalWeight = 0.6` but `progressedWeight = 20` — a 33x scale difference. `DistributionTargets` tried to fix this post-hoc but is currently disabled (`let scaledTokens = tokens`).
3. **Six layers of multiplication.** `baseWeight × priorityMultiplier × chartRulerMultiplier × WeightingModel.weight → token.weight → baseWeight × 2.0 + bonus + sunSignBoost → score × sunSignMultiplier`. No single place to see combined effect.

### What the new approach does differently

- **No string matching.** Planet-to-energy mapping uses typed enums and the `DailyFitCalibration.SignEnergyMap`.
- **Normalised weights at input.** `DailyFitCalibration.SourceWeights` sums to 1.0. Every astrological source (natal, transits, lunar, progressed, currentSun) contributes a proportional share.
- **One multiplication layer.** `rawContribution × sourceWeight × signMultiplier → energy score`. That's it.

### 3.1 Algorithm

The vibe profile is a 21-point distribution across 6 energies. The algorithm:

1. **Accumulate raw energy scores** from each astrological source, weighted by `calibration.sourceWeights`.
2. **Apply sun-sign multipliers** from `calibration.signEnergyMap`.
3. **Normalise to 21 integer points** using largest-remainder allocation.

#### Step 1: Raw score accumulation

For each of the 5 astrological sources, compute a contribution to each of the 6 energies:

**Natal contribution** (weight: `calibration.sourceWeights.natal`):
- For each planet in `natalChart.planets`, determine which energies it contributes to based on:
  - The planet's sign (e.g. Mars in Aries → Drama, Edge)
  - The planet itself (e.g. Venus always contributes some Romantic)
- Use `calibration.planetAxisMap` to look up planet-to-energy mappings. The existing `PlanetAxisMap` is planet-to-axis, so you will also need a `planetEnergyContribution` mapping. Define this as a private static dictionary within the engine:

```swift
private static let planetEnergyBase: [String: [Energy: Double]] = [
    "Sun":     [.drama: 0.3, .classic: 0.3, .playful: 0.2, .edge: 0.1, .romantic: 0.1],
    "Moon":    [.romantic: 0.4, .classic: 0.2, .playful: 0.2, .drama: 0.1, .utility: 0.1],
    "Mercury": [.playful: 0.3, .utility: 0.3, .classic: 0.2, .edge: 0.2],
    "Venus":   [.romantic: 0.4, .classic: 0.3, .playful: 0.2, .drama: 0.1],
    "Mars":    [.drama: 0.3, .edge: 0.3, .utility: 0.2, .playful: 0.2],
    "Jupiter": [.drama: 0.3, .playful: 0.3, .romantic: 0.2, .classic: 0.2],
    "Saturn":  [.classic: 0.4, .utility: 0.4, .drama: 0.1, .edge: 0.1],
    "Uranus":  [.edge: 0.5, .playful: 0.2, .drama: 0.2, .utility: 0.1],
    "Neptune": [.romantic: 0.4, .edge: 0.3, .drama: 0.2, .playful: 0.1],
    "Pluto":   [.drama: 0.4, .edge: 0.3, .romantic: 0.1, .utility: 0.1, .classic: 0.1]
]
```

Each planet's contribution is then modulated by its sign's elemental affinity:
- **Fire signs** (Aries, Leo, Sagittarius): boost Drama +0.1, Playful +0.05
- **Earth signs** (Taurus, Virgo, Capricorn): boost Classic +0.1, Utility +0.05
- **Air signs** (Gemini, Libra, Aquarius): boost Playful +0.1, Edge +0.05
- **Water signs** (Cancer, Scorpio, Pisces): boost Romantic +0.1, Drama +0.05

**Transit contribution** (weight: `calibration.sourceWeights.transits`):
- For each transit in `transits`, compute the transiting planet's energy base (same lookup), scaled by the transit's orb tightness (tighter orb = stronger influence).
- Hard transits (conjunction, square, opposition) amplify Drama and Edge.
- Soft transits (trine, sextile) amplify Romantic and Classic.

**Lunar phase contribution** (weight: `calibration.sourceWeights.lunarPhase`):
- Use `MoonPhaseInterpreter.Phase.fromDegrees(moonPhaseDegrees)` to get the phase.
- Map phases to energy biases:
  - New Moon → Utility, Classic (inward, foundational)
  - Waxing Crescent/First Quarter → Playful, Edge (building, experimenting)
  - Waxing Gibbous → Drama, Romantic (amplifying, expressing)
  - Full Moon → Drama, Playful (peak expression, visibility)
  - Waning Gibbous → Classic, Romantic (refining, appreciating)
  - Last Quarter/Waning Crescent → Utility, Edge (releasing, innovating)

**Progressed contribution** (weight: `calibration.sourceWeights.progressed`):
- Same as natal but using `progressedChart.planets`. The progressed chart moves slowly so this provides a gentle evolutionary drift.

**Current Sun contribution** (weight: `calibration.sourceWeights.currentSun`):
- Determine the current zodiac sign of the Sun based on `date`.
- Apply that sign's elemental energy bias as a background seasonal influence.
- This is the lightest contribution (~5%).

#### Step 2: Sun-sign multipliers

After accumulating raw scores across all sources:

```swift
let sunSign = extractSunSignName(from: natalChart)
for energy in Energy.allCases {
    let multiplier = calibration.signEnergyMap.multiplier(forSign: sunSign, energy: energy)
    rawScores[energy]! *= multiplier
}
```

This preserves the personality imprint of the user's sun sign. A Leo will always lean dramatic; a Virgo will always lean classic/utility. But the magnitude is controlled and bounded (multipliers are 0.85–1.5 in the default calibration).

#### Step 3: Normalise to 21 points

Use the **largest-remainder method** (same algorithm the legacy `normalizeToTwentyOne` uses, but cleaner):

1. Compute each energy's proportion: `proportion = rawScore / totalRawScore`.
2. Compute ideal allocation: `ideal = proportion × 21.0`.
3. Floor each to get integer base: `base = Int(ideal)`. Track remainder: `remainder = ideal - Double(base)`.
4. Distribute remaining points (21 - sum of bases) to energies with the largest remainders.
5. Clamp each energy to 0–10 range (the `VibeBreakdown` init already does this, but validate before constructing).
6. If clamping changed the total, redistribute the difference to unclamped energies.

## 4. Helper Functions You Will Need

```swift
private static func extractSunSignName(from chart: NatalChartCalculator.NatalChart) -> String
```
Find the Sun planet in `chart.planets` (where `planet.name == "Sun"`), get its `zodiacSign` (Int, 0–11), and convert to name string using `CoordinateTransformations.getZodiacSignName(sign:)`.

```swift
private static func signElement(forZodiacSign sign: Int) -> String
```
Returns "Fire", "Earth", "Air", or "Water" for the 12 signs (Aries=0 → Fire, Taurus=1 → Earth, etc.)

```swift
private static func currentSunSign(for date: Date) -> String
```
Approximate the Sun's zodiac sign from the date. Use the standard tropical date ranges. This does not need to be astronomically precise — it's the lightest source weight at 5%.
